# EKS Fargate: Cuándo Usar y Cuándo No

## Usar Fargate cuando

| Criterio | Descripción |
|----------|-------------|
| **Workloads event-driven** | Jobs esporádicos, cargas con peaks impredecibles que escalan a cero |
| **Microservicios pequeños/medianos** | Cada pod tiene sus propios recursos, sin overhead de gestión de nodos |
| **Compliance estricta** | Aislamiento total entre pods (cada pod su propia VM) |
| **No quieres gestionar infraestructura** | Sin necesidad de patch, upgrade o manage nodes |
| **Coste variable es aceptable** | Pay-per-use ideal si uso es irregular; costoso si uso es constante |

## No usar Fargate cuando

| Criterio | Descripción |
|----------|-------------|
| **Workloads stateful** | Bases de datos, storage persistente (Fargate no tiene storage local) |
| **Necesitas GPU o instancias especiales** | Fargate solo soporta CPU/memory configs limitados |
| **Cost optimization con uso continuo** | Si siempre tienes pods corriendo, nodos tradicionales son más baratos |
| **Requieres SSH/access directo** | No hay acceso a los nodos subyacentes |
| **Containers muy grandes** | Límites de 4vCPU/30GB memory por pod |
| **Requieres storages EBS persistentes** | Fargate no puede montar volúmenes EBS directamente |

## Regla práctica rápida

```
¿Spikes de carga impredecibles? → Fargate
¿Uso continuo 24/7?            → Node Groups
¿Stateful workload?             → Node Groups
¿Compliance/aislamiento máximo? → Fargate
```

## Resumen comparativo

| Aspecto | Fargate | Node Groups |
|---------|---------|-------------|
| Aislamiento | Total (cada pod en VM propia) | Compartido entre pods |
| Gestión | Serverless (sin gestión de nodos) | Manualmente o con Managed Node Groups |
| Storage | Solo EFS (network storage) | EBS, Instance Store, EFS |
| GPU/Especial | No | Sí |
| Límites pod | 4vCPU/30GB max | Depende del tipo de instancia |
| Coste | Pay-per-use (variable) | Pay-per-second fijo + uso constante |
| Acceso SSH | No | Sí |
| Stateful workloads | No recomendado | Soportado |

## Casos de uso típicos

### Fargate ✅
- APIs serverless con tráfico variable
- Workers de procesamiento de eventos (SQS, S3 events)
- Microservicios pequeños independientes
- Entornos de desarrollo/pruebas

### Node Groups ✅
- Aplicaciones con estado (databases)
- ML/AI workloads con GPUs
- Aplicaciones con alto consumo constante
- Requieren personalización de red o kernel
