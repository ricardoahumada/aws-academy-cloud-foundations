# Anexo 03b.02: Logs Insights Query Patterns

## Patrones Comunes de Queries

### Patrón 1: Errores por Período

```sql
fields @timestamp, @message
| filter @message like /(?i)error|exception|fail/
| stats count(*) as errorCount by bin(15m) as timeSlot
| sort timeSlot desc
```

**Casos de uso:**
- Monitoreo de tasa de errores en producción
- Alertas basadas en umbrales dinámicos
- Análisis de tendencias de errores

---

### Patrón 2: Top IPs con Más Errores

```sql
fields remoteAddr, @message
| filter @message like /error/i
| parse @message '* * * * * *' as ip1, ip2, ip3, ip4, ip5, rest
| stats count(*) as errorCount by remoteAddr
| sort errorCount desc
| limit 20
```

**Casos de uso:**
- Identificar posibles ataques DDoS
- Detectar scrapers abusivos
- Investigar fuentes de errores recurrentes

---

### Patrón 3: Latencia Percentil (p50, p95, p99)

```sql
fields @message
| parse @message '* * - - * * * * * * * * * * * * * *' as date, time, ip, user, method, path, protocol, status, size, referer, ua, duration, extra
| filter duration != "-"
| fields path, parseDouble(duration) as latencyMs
| stats 
    avg(latencyMs) as avgLatency,
    min(latencyMs) as minLatency,
    max(latencyMs) as maxLatency,
    percentile(latencyMs, 50) as p50,
    percentile(latencyMs, 95) as p95,
    percentile(latencyMs, 99) as p99
    by path
| sort p95 desc
```

**Casos de uso:**
- Identificar endpoints con latencia alta
- SLA monitoring
- Optimización de rendimiento

---

### Patrón 4: Análisis de Logs JSON Estructurados

```sql
fields @message
| filter @message like /^\{/
| parse @message '{ "level": "*", "message": "*", "timestamp": "*", "service": "*", "traceId": "*" }' as level, message, timestamp, service, traceId
| filter level = "ERROR"
| sort @timestamp desc
| limit 50
```

**Casos de uso:**
- Aplicaciones que emiten logs en JSON
- Correlación con traces de X-Ray
- Debugging distribuido

---

### Patrón 5: Usuarios con Más Solicitudes Fallidas

```sql
fields @message, @ingestionTime
| filter @message like /401|403|500/
| parse @message '* * * * * * * *' as date, time, user, method, path, status, size, duration
| filter status = "401" OR status = "403"
| stats count(*) as authFailures by user
| sort authFailures desc
| limit 10
```

**Casos de uso:**
- Detectar intentos de acceso no autorizado
- Identificar credenciales mal configuradas
- Auditoría de seguridad

---

### Patrón 6: Patrón de Accesos por Hora

```sql
fields @timestamp
| parse @timestamp '*-*-*T*:*:*Z' as year, month, day, hour, min, sec
| stats count(*) as requestCount by hour
| sort hour asc
```

**Casos de uso:**
- Análisis de tráfico por hora
- Planificación de capacidad
- Identificar horarios de mayor demanda

---

### Patrón 7: Encontrar Requests Lentos

```sql
fields @timestamp, @message
| filter @message like /latency/
| parse @message '*latency: *ms' as timestamp, latencyMs
| filter latencyMs > 1000
| sort latencyMs desc
| limit 20
```

**Casos de uso:**
- Identificar queries lentas
- Debugging de timeouts
- Optimización de BD

---

### Patrón 8: Correlación Errores + Traces

```sql
fields @message, @logStream
| filter @message like /exception/
| parse @message '*traceId: *' as prefix, traceId
| display traceId, @logStream, @message
| limit 20
```

**Casos de uso:**
- Relacionar errores con traces específicos
- Debugging end-to-end
- Investigación de incidentes

---

### Patrón 9: Análisis de Logs de Lambda

```sql
fields @message
| filter @message like /(?i)task timed out|connection error|throttling/
| parse @message 'Duration: * ms *' as duration, billedDuration
| stats 
    count(*) as timeoutCount,
    avg(billedDuration) as avgBilledDuration
    by bin(5m) as timeSlot
| sort timeSlot desc
```

**Casos de uso:**
- Identificar timeouts en funciones Lambda
- Detectar problemas de cold starts
- Monitorear límites de concurrencia

### Patrón 10: Dashboard Widget Queries

#### Errors con sparkline
```sql
fields @timestamp, @message
| filter @message like /error/i
| stats count(*) as errorCount by bin(1m) as minute
| sort minute asc
| limit 60
```

#### Status codes distribución
```sql
fields @message
| parse @message '* * - - * * * * *' as date, time, ip, user, method, path, protocol, status
| stats count(*) as count by status
| sort count desc
```

#### Top 5 slowest endpoints hoy
```sql
fields @timestamp, @message
| filter @message like / GET | POST | PUT | DELETE /
| parse @message '* * - - * * * * * * * *' as date, time, user, method, path, protocol, status, size, duration, extra
| fields method, path, avg(parseDouble(duration)) as avgDuration
| filter path like /\/api\//
| stats avg(avgDuration) as latency by method, path
| sort latency desc
| limit 5
```

## Funciones de Parseo Avanzadas

### Parseo con Regex

```sql
fields @message
| parse @message /(?<ip>\d+\.\d+\.\d+\.\d+).*(?:"(GET|POST|PUT|DELETE) (?<path>\S+))/
| filter method = "GET" AND path LIKE "/api/*"
| display ip, method, path, @timestamp
```

### Parseo Multi-Patrón

```sql
fields @message
| parse @message '* - - * * ["*" * *] * * * * * * * * *' 
    as date, time, user, method, path, protocol, status, size, referer, ua, duration, extra
| filter status >= 500
| sort @timestamp desc
```

### Extracción de JSON Profundo

```sql
fields @message
| filter @message like /^\{/
| parse @message '{"level": "*", "msg": "*", "ctx": {"userId": "*", "orgId": "*"}}'
    as level, msg, userId, orgId
| filter level = "error"
| stats count(*) by userId, orgId
| sort count desc
```

## Tips para Optimización

### Use LIMIT apropiadamente
Sin `LIMIT`, la query puede retornar miles de resultados innecesarios.

### Use bin() para agregaciones
`bin(15m)` agrupa resultados en intervalos de 15 minutos, más útil que timestamps exactos.

### Filter temprano
Aplica `FILTER` antes de `STATS` para reducir datos procesados.

### Usa @message solo cuando necesites
Si solo necesitas campos específicos, usa `FIELDS` para reducir transferencia de datos.

### Evita LIKE con patrones muy amplios
`@message like /./` escanea todo el mensaje. Sé específico.

## Comandos AWS CLI para Logs Insights

# Query en múltiples grupos de log
aws logs insights start-query \
    --log-group-names ["/aws/lambda/api", "/aws/lambda/auth"] \
    --start-time 1713500000 \
    --end-time 1713600000 \
    --query-string 'fields @timestamp, @message | filter @message like /error/i | limit 20'

    --log-group-name "/aws/lambda/api" \
    --max-results 10
