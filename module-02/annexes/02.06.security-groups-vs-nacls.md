# Security Groups vs NACLs - Comparativa

> **Tiempo de lectura:** ~8 min | **Referencia:** [AWS Security Groups](https://docs.aws.amazon.com/vpc/latest/userguide/security-groups.html) | [NACLs](https://docs.aws.amazon.com/vpc/latest/userguide/network-acls.html)

---

## Resumen Rápido

| Característica | Security Group | NACL |
|----------------|----------------|------|
| **Nivel** | Instancia | Subnet |
| **Estado** | **Stateful** | **Stateless** |
| **Reglas** | Solo `allow` | `allow` y `deny` |
| **Default** | Deny all | Allow all |
| **Evaluación** | Todas las reglas | Por orden (número) |
| **Aplicación** | Inmediata | Inmediata |

---

## Concepto: Stateful vs Stateless

### Security Group (Stateful)

**Regla:** La respuesta a tráfico permitido se permite automáticamente.

```
1.Inbound: SSH (puerto 22) desde 0.0.0.0/0 → PERMITIDO
2.Outbound: Respuesta SSH → AUTOMÁTICAMENTE PERMITIDA
3.No necesitas regla outbound para SSH reply
```

### NACL (Stateless)

**Regla:** La respuesta NO se permite automáticamente. Debes crear regla explícita.

```
1.Inbound: SSH (puerto 22) desde 0.0.0.0/0 → PERMITIDO
2.Outbound: Respuesta SSH → DEBES PERMITIR EXPLÍCITAMENTE
3.Sin regla outbound → respuesta BLOQUEADA
```

---

## Ejemplo Práctico: Permitir HTTP

### Con Security Group

```
Inbound:  TCP 80  0.0.0.0/0   → Allow
Outbound: (default allow all) → No necesitas regla
Resultado: ✅ Respuesta HTTP funciona
```

### Con NACL

```
Inbound:  100  TCP 80  0.0.0.0/0  → Allow
Outbound: 100  TCP 1024-65535  0.0.0.0/0  → Allow (respuesta)

⚠️ SIN outbound: La página no carga porque la respuesta está bloqueada
```

---

## Orden de Evaluación

```
Internet ──► NACL (Stateless, evaluated FIRST)
                │
                ▼ (si NACL permite)
           Security Group (Stateful, evaluated SECOND)
                │
                ▼ (si SG permite)
            EC2 Instance
```

**Importante:** Si el NACL deniega, el Security Group nunca ve el tráfico.

---

## Número de Reglas en NACL

| Número | Regla | Efecto |
|--------|-------|--------|
| 100 | Allow TCP 80 | HTTP permitido |
| 110 | Allow TCP 443 | HTTPS permitido |
| 120 | Allow TCP 22 | SSH permitido |
| **\* | Deny TCP 80 from 203.0.113.0/24 | Bloquear IP específica |
| 999 | Deny ALL | Bloquear todo lo demás |

**Menor número = evaluado primero**

Para bloquear una IP específica, usa número menor que tus reglas allow (ej: 90).

## Cuándo Usar Cada Uno

| Capa | Herramienta | Uso |
|------|-------------|-----|
| **Subnet** | NACL | Bloquear rangos específicos, deny explícito, reglas temporales |
| **Instancia** | Security Group | Permitir solo tráfico necesario por puerto/aplicación |

### Recomendación

**Layer 1 (NACL - subnet):**
- Permitir HTTP/HTTPS desde cualquier lugar
- Permitir SSH solo desde tu VPC (10.0.0.0/16)
- Denegar tráfico específico conocido como malicioso

**Layer 2 (Security Group - instancia):**
- Permitir HTTP (80) desde 0.0.0.0/0
- Permitir HTTPS (443) desde 0.0.0.0/0
- Permitir SSH (22) desde tu IP específica

## Comandos AWS CLI

```bash
# Crear Security Group
aws ec2 create-security-group \
  --group-name mi-sg-web \
  --description "Security Group para web server" \
  --vpc-id vpc-xxxx

# Agregar regla inbound
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxx \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0

# Agregar regla para SSH desde VPC
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxx \
  --protocol tcp \
  --port 22 \
  --cidr 10.0.0.0/16

# Crear NACL
aws ec2 create-network-acl --vpc-id vpc-xxxx

# Agregar regla a NACL (Inbound)
aws ec2 create-network-acl-entry \
  --network-acl-id acl-xxxx \
  --rule-number 100 \
  --protocol tcp \
  --rule-action allow \
  -- egress false \
  --port-range From=80,To=80 \
  --cidr-block 0.0.0.0/0

# Agregar regla a NACL (Outbound para respuesta)
aws ec2 create-network-acl-entry \
  --network-acl-id acl-xxxx \
  --rule-number 100 \
  --protocol tcp \
  --rule-action allow \
  --egress true \
  --port-range From=1024,To=65535 \
  --cidr-block 0.0.0.0/0
```

## Errores Comunes

| Error | Causa | Solución |
|-------|-------|----------|
| Conexión no responde | NACL no tiene regla outbound | Agregar outbound para respuesta |
| No puedo bloquear IP | NACL default allow all | Agregar regla deny con número menor |
| SSH no funciona | SG no permite puerto 22 | Agregar regla inbound puerto 22 |
| Depuración difícil | Mezcla de NACL y SG | Revisar primero NACL, luego SG |
| Reglas contradictorias | NACL deniega lo que SG permite | NACL evaluated primero, revierte orden |

---

## Best Practices

1. **NACL:** Usar para crear un perímetro de seguridad en la subnet
2. **Security Groups:** Usar como firewall principal por instancia
3. **Nunca依赖-default NACLs:** Crear NACLs específicos si necesitas deny
4. **Orden de reglas:** Poner reglas de deny antes en NACL (números menores)
5. **Docum entar cambios:** Cada vez que modifiques, documentar el cambio
