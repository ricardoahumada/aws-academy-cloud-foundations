# VPC Peering - Guía de Referencia

> **Tiempo de lectura:** ~6 min | **Referencia:** [AWS VPC Peering Documentation](https://docs.aws.amazon.com/vpc/latest/peering/)

---

## ¿Qué es VPC Peering?

Es una conexión de red privada entre dos VPCs usando la infraestructura de AWS.

```
┌─────────────────┐          ┌─────────────────┐
│      VPC A      │          │      VPC B      │
│   10.0.0.0/16   │◄────────►│  172.16.0.0/16  │
│                 │ peering  │                 │
│  10.0.1.0/24    │          │  172.16.1.0/24  │
└─────────────────┘          └─────────────────┘
```

**Beneficios:**
- Latencia baja (tráfico interno AWS)
- Ancho de banda alto (hasta 100 Gbps)
- Sin costo por tráfico entre regiones
- Sin NAT, sin VPN

---

## Características Principales

| Característica | Detalle |
|----------------|---------|
| Latencia | Igual que tráfico interno AWS |
| Ancho de banda | Hasta 100 Gbps |
| Costo | Gratis entre regiones AWS |
| CIDRs | NO deben solaparse |
| Compatible | IPv4 e IPv6 |

---

## ⚠️ Limitación Crítica: No Transitivo

**El problema más común con Peering es entender que NO es transitivo.**

```
Escenario:
  VPC A (10.0.0.0/16) ◄──► VPC B (172.16.0.0/16)
                            │
                            ▼
                         VPC C (192.168.0.0/16)

❌ A NO puede comunicarse con C a través de B
❌ B NO puede re-enviar tráfico de A hacia C

→ Debes crear un Peering directo A ↔ C
```

### Analogía

```
Peering = Puente directo entre dos ciudades

VPC A ◄──► VPC B     (puente A-B existe)
                │
                ▼
             VPC C     (no hay puente B-C para A)
```

---

## Casos de Uso

| Escenario | ¿Peering recomendado? | Alternativa |
|-----------|------------------------|-------------|
| Dos VPCs que necesitan comunicarse | ✅ Sí | - |
| Tres VPCs en cadena A→B→C | ⚠️ Cuidado, crear A↔C | Transit Gateway |
| Más de 4 VPCs conectadas | ❌ No | Transit Gateway |
| Conectar a red on-premise | ❌ No | VPN o Direct Connect |

### Regla Simple

| Número de VPCs | Recomendación |
|----------------|---------------|
| 2 | Peering |
| 3-4 | Peering (crear mesh parcial) |
| 5+ | Transit Gateway |

---

## Comparativa: Peering vs Transit Gateway

| Característica | Peering | Transit Gateway |
|----------------|---------|-----------------|
| Conexiones necesarias | N*(N-1)/2 | N |
| Transitivo | ❌ No | ✅ Sí |
| Administración | Complex si N>3 | Centralizado |
| Costo | Gratis | Por hora + tráfico |
| Casos uso | 2-4 VPCs | 5+ VPCs |

**Ejemplo con 5 VPCs:**
- Peering: 5*(5-1)/2 = **10 conexiones**
- Transit Gateway: **5 conexiones**

## Proceso de Creación

```
┌─────────────────────────────────────────────────────────────────┐
│ Paso 1: Crear Request (VPC A)                                  │
│ ─────────────────────────────────────────────────────────────── │
│ Owner de VPC A inicia:                                          │
│   - Selecciona VPC B como acceptor                              │
│   - AWS genera peering connection ID: pcx-xxxxx                │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ Paso 2: Aceptar Request (VPC B)                                 │
│ ─────────────────────────────────────────────────────────────── │
│ Owner de VPC B acepta:                                           │
│   - Ir a VPC > Peering Connections                             │
│   - Seleccionar pcx-xxxxx > Actions > Accept Request            │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ Paso 3: Actualizar Route Tables (AMBOS lados)                    │
│ ─────────────────────────────────────────────────────────────── │
│ En VPC A:                                                        │
│   Destination: 172.16.0.0/16 → Target: pcx-xxxxx                 │
│                                                                  │
│ En VPC B:                                                        │
│   Destination: 10.0.0.0/16 → Target: pcx-xxxxx                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ Paso 4: Verificar Security Groups                               │
│ ─────────────────────────────────────────────────────────────── │
│ En cada VPC, el SG de las instancias debe permitir tráfico       │
│ desde el CIDR de la otra VPC si es necesario.                   │
└─────────────────────────────────────────────────────────────────┘
```

## Comandos AWS CLI

```bash
# Crear request de Peering
aws ec2 create-vpc-peering-connection --vpc-id vpc-aaaaaa \
  --peer-vpc-id vpc-bbbbbb

# Aceptar Peering (en la cuenta del acceptor)
aws ec2 accept-vpc-peering-connection --vpc-peering-connection-id pcx-xxxx

# Ver estado del Peering
aws ec2 describe-vpc-peering-connections \
  --vpc-peering-connection-ids pcx-xxxx

# Agregar ruta en Route Table (VPC A)
aws ec2 create-route --route-table-id rt-aaaaaa \
  --destination-cidr-block 172.16.0.0/16 \
  --vpc-peering-connection-id pcx-xxxx

# Agregar ruta en Route Table (VPC B)
aws ec2 create-route --route-table-id rt-bbbbbb \
  --destination-cidr-block 10.0.0.0/16 \
  --vpc-peering-connection-id pcx-xxxx

# Eliminar Peering
aws ec2 delete-vpc-peering-connection --vpc-peering-connection-id pcx-xxxx
```

## Errores Comunes

| Error | Causa | Solución |
|-------|-------|----------|
| `InvalidCIDRBlock` | CIDRs se solapan | Usar CIDRs no superpuestos |
| Timeout en ping | Falta ruta en RT | Agregar ruta en ambos lados |
| Conexión拒绝 | Security Group bloquea | Permitir tráfico en SG |
| Notransitivo | Expectativa incorrecta | Crear Peering directo A↔C |
